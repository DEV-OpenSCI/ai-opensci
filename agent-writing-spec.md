# Agent 书写规范

> 版本：v1.0 | 适用范围：所有 AI Agent 的 Markdown 配置文件（CLAUDE.md / AGENT.md / SYSTEM.md 等）

---

## 一、文件定位

Agent 的 `.md` 文件是**给 AI 的持久化操作手册**，不是给人类阅读的说明文档。  
它的质量直接决定 Agent 的执行质量。核心目标：

- 让 Agent 在无人监督时做出正确判断
- 减少歧义，消除"自由发挥"空间
- 覆盖边界场景，不只写 Happy Path

---

## 二、文件结构规范

每份 Agent `.md` 文件必须包含以下章节，顺序不做强制要求，但需完整：

```
1. 角色定义     ← 我是谁，职责边界在哪
2. 项目背景     ← 所在系统/项目的关键上下文
3. 行为规则     ← 必须做 / 禁止做 / 默认行为
4. 工具使用     ← 哪些工具，何时用，何时不用
5. 输出格式     ← 语言、结构、详略程度
6. 错误处理     ← 异常情况的应对策略
7. 优先级声明   ← 规则冲突时的取舍顺序
```

## 三、语言风格规范

### 3.1 使用命令式动词

```markdown
# ❌ 不推荐（描述性）
Claude 是一个尽量准确、乐于助人的助手。

# ✅ 推荐（命令式）
- 回答前先验证信息时效性
- 不确定时明确说明，不得编造内容
```

### 3.2 禁止使用模糊词汇

以下词汇会导致 Agent 无法判断边界，**一律替换为可量化的条件**：

| 禁用词 | 替换方向 |
|--------|---------|
| 尽量、适当 | 明确阈值或条件 |
| 有时候、偶尔 | 明确触发场景 |
| 尽可能 | 给出优先级排序 |
| 差不多 | 给出具体数值范围 |

```markdown
# ❌ 不推荐
尽量保持回复简洁。

# ✅ 推荐
回复长度规则：
- 简单问答：< 100 字
- 技术解释：< 500 字
- 完整报告：不限，但需要有摘要段落
```

### 3.3 正反例并列

重要规则必须同时给出正例和反例：

```markdown
# 文件命名规则
- ✅ feature-user-auth.ts
- ✅ utils-date-formatter.ts
- ❌ Feature_UserAuth.ts（不用大写和下划线混合）
- ❌ utils.ts（过于宽泛，不说明用途）
```

## 四、内容规范

### 4.1 角色定义

必须包含：
- 具体职责（不是"助手"，而是"负责 XX 模块的代码审查"）
- 明确的职责边界（我不做什么）
- 上游和下游关系（我从哪里接收任务，交付给谁）

```markdown
## 角色定义
你是 OPENSCI 项目的智能合约审计 Agent。

**职责：**
- 对提交的 Solidity 代码进行安全漏洞扫描
- 输出结构化审计报告（JSON 格式）
- 对高危漏洞发出阻断信号，阻止部署流程继续

**不在职责内：**
- 不修改代码（只标注问题）
- 不做业务逻辑评审（只关注安全性）
- 不与外部系统通信（无网络工具权限）
```

### 4.2 行为规则

分三类写，缺一不可：

```markdown
## 行为规则

### 必须做（强制）
- 每次输出前检查格式是否符合 output_schema
- 发现高危漏洞时必须终止流程并上报

### 禁止做（红线）
- 禁止在输出中包含原始私钥或助记词
- 禁止对未经授权的合约地址发起链上调用
- 禁止忽略 CRITICAL 级别的 lint 错误

### 默认行为（无明确指令时）
- 默认使用中文输出
- 默认对所有外部输入做合法性校验
- 默认保存中间结果到 ./tmp/agent_state.json
```

### 4.3 工具使用

每个工具必须说明：使用场景、禁用场景、输出处理方式。

```markdown
## 工具使用

| 工具 | 触发条件 | 禁用条件 | 输出处理 |
|------|---------|---------|---------|
| web_search | 需要 2024 年后的信息 | 纯逻辑推导、代码问题 | 摘要后引用，不直接转述 |
| code_executor | 需要验证代码输出 | 生产环境敏感操作 | 捕获 stderr，异常时中止 |
| file_write | 输出超过 50 行 | 覆盖 .env / config.prod.* | 写入前确认路径合法性 |
```

### 4.4 输出格式

```markdown
## 输出格式

**语言：** 用户使用中文提问 → 中文回答；代码注释统一用英文

**结构：**
- 结论前置，先给答案再给推导过程
- 代码必须附文件路径和用途说明
- 超过 3 个步骤的操作必须用编号列表

**不确定时的表达：**
- 使用"可能"、"建议验证"等措辞
- 不得以肯定语气输出未经验证的信息
- 置信度低于 70% 时，主动说明局限性
```

### 4.5 错误处理

```markdown
## 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| 工具调用失败 | 重试 1 次，仍失败则记录错误并跳过该步骤 |
| 输入格式不合法 | 拒绝执行，返回具体的格式错误说明 |
| 输出验证不通过 | 重新生成，最多 2 次，超限则标记为 needs_review |
| 超出 context 限制 | 压缩历史摘要，保留最近 3 轮 + 初始指令 |
| 遇到未定义场景 | 暂停执行，输出 {status: "blocked", reason: "..."} |
```

### 4.6 优先级声明

规则冲突时 Agent 必须有明确的取舍依据：

```markdown
## 优先级（从高到低）

1. **安全性** — 任何可能造成资产损失或数据泄露的操作，无条件中止
2. **正确性** — 宁可不输出，也不输出错误信息
3. **完整性** — 尽量覆盖任务的所有要求
4. **简洁性** — 在满足以上条件后，保持输出精炼
5. **速度**   — 最低优先级，不以牺牲质量换取速度
```

## 五、长度与密度控制

| 章节 | 建议长度 |
|------|---------|
| 核心规则（角色 + 行为 + 优先级） | ≤ 500 tokens |
| 工具使用说明 | ≤ 300 tokens |
| 输出格式 | ≤ 200 tokens |
| 错误处理 | ≤ 200 tokens |
| 完整文件总计 | ≤ 1500 tokens（精华版）/ ≤ 4000 tokens（完整版） |

> 超出限制的内容移入 `AGENT_APPENDIX.md`，主文件通过 `@import` 或链接引用。

## 六、版本管理规范

```markdown
# 文件头部必须包含版本信息
---
agent: <agent_name>
version: 1.2.0
updated: 2025-03-05
owner: <team/person>
---
```

**版本号规则：**
- `Major`：角色定义或职责边界变更
- `Minor`：新增规则或工具
- `Patch`：措辞修正、修复 Bad Case

**更新触发条件：**
- 每次发现 Agent 的 Bad Case，必须回溯并补充规则
- 每次新增工具或接口变更，同步更新工具使用章节

---

## 七、Bad Case 驱动改进流程

```
Agent 产生 Bad Case
        ↓
分析根因：规则缺失 / 规则模糊 / 规则冲突？
        ↓
        ├─ 缺失 → 在对应章节补充规则
        ├─ 模糊 → 用量化条件替换模糊描述
        └─ 冲突 → 在优先级章节明确取舍顺序
        ↓
更新版本号（Patch 或 Minor）
        ↓
回归测试：用原 Bad Case 场景验证
```
