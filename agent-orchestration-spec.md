# Agent 编排规范

> 版本：v1.0 | 适用范围：所有涉及多 Agent 协作的系统设计与实施

---

## 一、核心原则

> **把不确定性（LLM）关在最小的盒子里，用确定性逻辑控制流程。**

| 原则 | 含义 |
|------|------|
| **最小权限** | 每个 Agent 只拥有完成本职任务所需的最小工具和信息 |
| **显式优于隐式** | 所有依赖、路由、状态转换必须在编排图中显式声明 |
| **可观测** | 每一步执行都必须可追踪、可回放 |
| **可中断** | 任意节点失败后，系统能从断点恢复，不必全量重跑 |
| **人工兜底** | 关键决策节点必须有人工介入入口 |

---

## 二、编排架构设计

### 2.1 标准分层结构

```
┌─────────────────────────────────────┐
│          Orchestrator Layer          │  ← 任务拆解、路由分发、结果聚合
├──────────────┬──────────────────────┤
│   Agent A    │   Agent B   │Agent C  │  ← 执行层，单一职责
├──────────────┴──────────────────────┤
│              Tool Layer             │  ← 工具调用（API / DB / 文件系统）
├─────────────────────────────────────┤
│           State Store               │  ← 共享状态持久化层
└─────────────────────────────────────┘
```

### 2.2 三种通信模式

**模式一：顺序链 Sequential**
```
Task → Agent A → Agent B → Agent C → Output
```
- 适用：强依赖，前序结果影响后续执行
- 要求：每步设置 checkpoint，支持从任意节点重试

**模式二：并行扇出 Parallel Fan-out**
```
Task ──┬→ Agent A ──┐
       ├→ Agent B ──┼→ Aggregator → Output
       └→ Agent C ──┘
```
- 适用：子任务相互独立，可同时执行
- 要求：聚合层必须处理部分失败场景

**模式三：路由分发 Router**
```
Task → Router → [条件 A] → Agent A
              → [条件 B] → Agent B
              → [条件 C] → Agent C
```
- 适用：不同类型任务走不同处理路径
- 要求：路由逻辑用确定性代码实现，不依赖 LLM 判断

## 三、任务拆解规范

### 3.1 粒度判断标准

一个子任务应满足以下条件才算粒度合适：

- [ ] 单个 Agent 的 context window 能舒适容纳所需信息（建议 < 60% 上限）
- [ ] 该子任务可以独立测试、独立失败、独立重试
- [ ] 子任务边界清晰，不与其他子任务有隐式状态共享
- [ ] 输入和输出格式可以用 schema 完整描述

### 3.2 拆解粒度对照表

| 信号 | 说明 | 处理方式 |
|------|------|---------|
| 单 Agent Token 持续超 8000 | 任务太重 | 继续拆分 |
| Agent 间消息来回超 5 次 | 边界不清 | 重新定义职责 |
| 子任务数量 > 10 | 编排太复杂 | 考虑分层编排 |
| 子任务执行时间差异 > 10x | 存在瓶颈节点 | 拆分或异步化 |

## 四、接口设计规范

### 4.1 Agent 间消息格式（强制）

所有 Agent 间传递的消息必须是结构化数据，禁止传递纯自然语言段落。

```json
{
  "task_id": "task-20250305-001",
  "from": "orchestrator",
  "to": "audit-agent",
  "type": "execute",
  "payload": {
    "input": { ... },
    "context": { "只传必要字段": true }
  },
  "metadata": {
    "created_at": "2025-03-05T10:00:00Z",
    "timeout_seconds": 120,
    "retry_count": 0
  }
}
```

### 4.2 Agent 响应格式（强制）

```json
{
  "task_id": "task-20250305-001",
  "status": "success | failed | blocked | needs_review",
  "output": { ... },
  "error": {
    "code": "TOOL_TIMEOUT",
    "message": "...",
    "retryable": true
  },
  "metadata": {
    "tokens_used": 1250,
    "duration_ms": 3200,
    "model": "claude-sonnet-4"
  }
}
```

### 4.3 信息蒸馏原则

每一层传递都必须做"信息浓缩"，而不是全量透传：

```
# ❌ 禁止：直接传递上游完整输出
agent_b.run(input=agent_a.full_output)

# ✅ 要求：提取关键字段后传递
agent_b.run(input=extract_key_fields(agent_a.output))
```

## 五、状态管理规范

### 5.1 状态必须外化

**禁止**依赖 Agent 的对话历史来传递状态。  
**要求**所有中间状态写入共享存储：

```
共享存储选型参考：
- 本地/单机：  文件系统（JSON/SQLite）
- 分布式：     Redis / PostgreSQL
- 云原生：     对象存储（S3/OSS）+ 状态数据库
```

### 5.2 标准状态机定义

每个任务节点必须有明确的状态定义：

```
PENDING     → 已创建，等待执行
RUNNING     → 执行中
SUCCESS     → 执行成功，输出可用
FAILED      → 执行失败
  ├─ RETRYABLE   → 可自动重试
  └─ FATAL       → 需人工介入
BLOCKED     → 等待外部条件（人工审核、依赖任务）
SKIPPED     → 被编排器跳过（降级处理）
```

### 5.3 Checkpoint 规范

```
设置 Checkpoint 的时机：
- 每个主要阶段完成后
- 调用外部系统（API / 链上操作）前后
- 用户可见的状态变更点

Checkpoint 内容：
- 当前阶段标识
- 已完成节点的输出摘要
- 下一步执行所需的最小上下文
- 时间戳 + 版本号
```

### 5.4 幂等性要求

每个 Agent 节点必须设计为幂等操作：

- 相同输入多次执行，结果一致
- 副作用操作（写文件、链上交互）必须有去重机制
- 推荐使用 `task_id` 作为幂等键

## 六、错误处理规范

### 6.1 错误分级策略

```
Level 1 — 自动重试
  触发条件：网络超时、Rate Limit、临时性工具失败
  处理策略：指数退避重试，最多 3 次（1s → 2s → 4s）
  超限后：升级为 Level 2

Level 2 — 降级处理
  触发条件：非关键子任务失败，不阻塞主流程
  处理策略：跳过该节点 + 标记 SKIPPED + 继续执行
  记录：降级原因写入任务日志

Level 3 — 人工介入
  触发条件：关键节点失败 / 输出置信度过低 / 涉及资产操作
  处理策略：暂停整个任务 → 通知责任人 → 等待人工确认
  超时未处理：72h 后自动关闭任务

Level 4 — 全链路回滚
  触发条件：多个关键节点连续失败 / 检测到数据污染
  处理策略：回滚到最近 Checkpoint → 记录完整错误链 → 通知
```

### 6.2 置信度校验节点

在关键节点之间插入验证层：

```
[执行 Agent] → [验证器] → [下一节点]
                  ↓
             不通过 → 重新执行（最多 2 次）
                  ↓
             仍不通过 → 标记 needs_review，通知人工
```

验证器检查内容：
- 输出格式是否符合 schema
- 关键字段是否存在且合法
- 数值是否在合理范围内
- 与上下文是否存在明显矛盾

## 七、终止条件规范

### 7.1 必须显式设置的终止条件

```python
# 每个编排任务必须设置以下上限
MAX_ITERATIONS    = 10        # 最大循环/迭代次数
MAX_TOKENS_TOTAL  = 100_000   # 全链路 token 预算
TIMEOUT_SECONDS   = 600       # 总超时时间
MAX_RETRIES       = 3         # 单节点最大重试次数
MAX_COST_USD      = 5.0       # 最大费用上限（按需设置）
```

### 7.2 完成判断标准

不得让 Agent 自行决定"任务是否完成"，必须有明确的完成判断逻辑：

```
# ❌ 禁止
"当你认为任务完成时，停止执行"

# ✅ 要求
完成条件（满足任一）：
- output_schema 验证通过
- 所有必填字段均已填充
- 外部系统确认回调已收到
```

## 八、可观测性规范

### 8.1 必须记录的信息

| 记录项 | 格式 | 保留期 |
|--------|------|--------|
| 每个节点的输入摘要 | JSON，截断至 500 字符 | 30 天 |
| 每个节点的输出摘要 | JSON，截断至 500 字符 | 30 天 |
| 路由决策及依据 | 枚举值 + 条件描述 | 30 天 |
| Token 消耗 & 耗时 | 数值 | 90 天 |
| 错误信息 & 重试记录 | 结构化 | 90 天 |
| 人工介入记录 | 操作人 + 操作内容 + 时间 | 永久 |

### 8.2 Trace 结构

```json
{
  "trace_id": "trace-xxx",
  "task_id": "task-xxx",
  "spans": [
    {
      "span_id": "span-001",
      "agent": "orchestrator",
      "action": "route",
      "start_time": "...",
      "duration_ms": 50,
      "decision": "→ audit-agent",
      "reason": "input.type == 'solidity'"
    }
  ]
}
```

### 8.3 工具推荐

| 规模 | 推荐方案 |
|------|---------|
| 本地 / 小团队 | 结构化 JSON 日志 + 文件存储 |
| 中等规模 | LangFuse / LangSmith |
| 生产级 | OpenTelemetry + 自建 trace pipeline |

## 九、安全与权限规范

### 9.1 最小权限原则

```
每个 Agent 只能访问：
- 完成本任务所需的工具（不多给）
- 本任务上下文所需的数据（不多传）
- 本任务执行所需的时间窗口（不长期持有）
```

### 9.2 危险操作保护

以下操作必须经过人工确认，不得由 Agent 自主执行：

- 链上资产转移 / 合约部署
- 生产数据库写操作
- 外部 API 的非幂等请求（POST / DELETE）
- 系统权限变更

### 9.3 沙箱隔离

- Agent 的代码执行环境必须沙箱隔离
- 禁止 Agent 直接访问宿主机文件系统（通过路径白名单控制）
- 网络访问按域名白名单控制

## 十、编排设计 Checklist

在上线任何多 Agent 编排系统前，逐条确认：

**架构层面**
- [ ] 每个 Agent 的职责边界是否清晰且互不重叠？
- [ ] Agent 间的接口（输入/输出格式 & schema）是否已定义？
- [ ] 路由逻辑是否用确定性代码实现（不依赖 LLM 判断）？
- [ ] 是否有分层编排而非单一 God Orchestrator？

**状态层面**
- [ ] 是否有状态持久化机制？
- [ ] 是否定义了完整的状态机？
- [ ] 关键节点是否设置了 Checkpoint？
- [ ] 所有节点是否满足幂等性要求？

**容错层面**
- [ ] 是否定义了所有错误类型的处理策略？
- [ ] 是否设置了终止条件和资源上限（Token / 时间 / 费用）？
- [ ] 是否有完成判断的明确标准？
- [ ] 关键节点是否有置信度校验？

**运营层面**
- [ ] 是否有完整的可观测性（日志 / Trace）？
- [ ] 关键决策节点是否有人工审核入口？
- [ ] 整个流程是否可以从任意断点恢复？
- [ ] 危险操作是否有权限保护和人工确认？

## 十一、常见反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| **God Orchestrator** | 中心 Agent 决定一切，成为瓶颈 | 分层编排，局部自治 |
| **上下文污染** | 历史对话越来越长，质量下降 | 每个 Agent 新开对话，只传结构化摘要 |
| **隐式依赖** | Agent 通过副作用传递状态 | 所有依赖显式声明在编排图中 |
| **过度信任 LLM 路由** | 让 LLM 决定"下一步做什么" | 确定性逻辑控制主流程 |
| **无终止条件** | 循环 Agent 无限运行 | 强制设置迭代上限和超时 |
| **没有人工介入点** | 全自动链路跑偏无法干预 | 关键节点设置 human-in-the-loop |
| **信息全量透传** | 每层都传递完整上下文 | 每层做信息蒸馏，只传必要字段 |
